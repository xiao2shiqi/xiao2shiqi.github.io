用了 AI 编程工具这么久，我发现一件很尴尬的事。

大多数人其实不是在用它编程，而是在用它写代码。看起来很像，但本质不同。

写代码是：你知道要做什么，你只是让它帮你打字。

编程是：你要决策架构、排查问题、控制风险、把一堆不确定性变成可交付的东西。

用 AI 工具“打字”很容易，但真正的工程问题，比如接手一个陌生系统、做一次跨模块重构、排查一个线上诡异 bug，很多人还是卡壳。不是工具不够强，是根本不知道应该往哪个方向问、怎么问、问完之后怎么推进。

最近我看了 OpenAI 公开的一份 PDF：《How OpenAI uses Codex》。这份文档讲的是他们自己的工程师怎么在日常工作里使用 Codex，覆盖了 7 类高频场景，以及他们沉淀下来的最佳实践。

原文链接：[How OpenAI uses Codex](https://cdn.openai.com/pdf/6a2631dc-783e-479b-b1a4-af0cfbd38630/how-openai-uses-codex.pdf)

读完之后我最大的感受是：他们并没有把 Codex 当成一个更聪明的自动补全，而是把它当成工程流程里的一个“标准件”，嵌进了真实的工作节奏里。

这篇文章我把这份 PDF 拆了一遍，用我自己的语言整理出两件事：他们在哪 7 个场景里高频使用 Codex，以及他们总结出来的最佳实践是什么。你可以直接对照自己的工作方式，看哪里能搬过来用。

### OpenAI 的 7 个高频用法

#### 代码理解：把陌生系统变成“可提问的东西”

接手一个陌生系统，最折磨人的不是代码量，是那种“不知道从哪下手”的感觉。

不知道核心逻辑在哪，不知道模块之间怎么依赖，不知道一个请求进来之后到底走了哪条路。这种情况下你甚至没办法开始工作，更别说 oncall 时要在压力下快速定位。

OpenAI 的工程师用 Codex 做的第一件事，往往不是写代码，而是“问系统”：认证逻辑在哪、请求链路是什么、哪个模块如果出问题会影响哪里。那些“理应存在但没人写”的文档，也顺手让它生成。

还有一个细节很值得学：修完 bug 之后不是关掉对话，而是接着问：“同类问题还可能出现在哪？”这一步，把一次修复变成了一次预防性扫描。

#### 重构与迁移：一致性才是真正的难点

重构最难的不是“改”，是“改得一致”。

你改了这个文件里的 API 调用，但仓库里还有另外八个地方用的是旧写法。你改完上线，线上炸了，然后才发现漏了一处。这种事情发生过一次，你就会对跨文件改动产生本能恐惧。

OpenAI 的工程师处理这类任务的方式是：先扫描全仓库，找出所有受影响的地方，把影响范围总结出来，然后再动手。不是蒙眼改，而是先弄清楚边界。

从拆分大模块、替换旧模式，到把紧耦合代码改成可测结构，Codex 做的是把一致性工作批量化，而不是帮你逐个文件手动改。

#### 性能优化：先找到真正在痛的地方

性能问题最坑的一点是：你以为在慢的地方，往往不是真正慢的地方。

你去优化一个“看起来慢”的循环，但真正瓶颈可能是另一个每次请求都反复执行的昂贵操作，或者是一段没加缓存的 DB 查询。肉眼 review 很难发现这些，因为你不知道它在生产上到底跑了多少次。

OpenAI 文档里有工程师愿意花 5 分钟写一个 prompt，换来省下 30 分钟手工排查。这笔账很好算。

把慢路径、高内存代码丢给 Codex 分析，让它告诉你问题在哪、替代方案是什么，你再上手决策。节省的不只是时间，还有“凭感觉猜测”的认知消耗。

#### 提升测试覆盖：把“应该有但没做”的事情补回来

写测试这件事，大多数工程师的真实状态是：能过就行，边界条件能省就省。

不是不知道应该覆盖，是真的没时间，或者没精力。结果就是低覆盖模块永远低覆盖，直到某天出了事故才发现“这里没有任何测试”。

OpenAI 的工程师把 Codex 直接指向那些覆盖率低的模块，让它批量生成可运行的单测 PR，包括边界条件和失败路径。甚至可以过夜跑一轮，第二天只需要 review。

这件事之所以以前做不了，不是因为没人想做，是因为“生成一批有质量的测试”本身就是一件费时费力的事。现在这件事的成本变了。

#### 提升研发速度：开工时搭架子，收尾时填坑

开工阶段最浪费时间的，不是写业务逻辑，而是搭那些“每次都差不多”的脚手架：目录结构、模块初始化、API stub、基础日志与校验。

临近发布时最难受的是“杂而关键”的尾部工作：bug triage、监控埋点、配置核查、把产品反馈转成初稿代码。每件都不大，加在一起就是一个下午。

OpenAI 的工程师把这两端都交给 Codex：开工交给它搭架子，收尾交给它填坑。你的核心精力就能留在真正需要判断的部分。

#### 保持心流：让 Codex 替你“占住任务”

会议多的工作日，是最消耗工程师的。

你上午在一个问题上进入状态了，下午要开三个会。你知道有几件事顺手可以做，但你不想切换上下文，因为一旦切换，回来再捡至少半小时。于是这几件事一直悬在脑子里，既没做，又在消耗注意力。

OpenAI 的工程师处理方式很直接：想到就丢给 Codex 一个任务，让它生成计划或先跑框架。你去开会，Codex 在跑。你回来时，那边已经有东西可以 review 了，不需要重建上下文。

他们还有一个做法：把 Slack 线程、trace、issue 这些零散材料汇总丢给 Codex，下午捡回来的时候，直接从整理好的状态继续。

#### 探索与方案构思：让它帮你“反对你自己”

有一种卡壳不是不知道怎么做，而是你有想法，但不确定是否靠谱。

你觉得当前设计有问题，但说不清具体在哪。你在两个方案之间犹豫，脑子里两边论据混在一起。你想知道换一种架构会怎样，但推演需要时间。

OpenAI 的工程师会把这类开放问题直接丢进去：换成事件驱动会怎样、哪些假设值得质疑、这个决策的 tradeoff 是什么。让它把你没想到的角度列出来，帮助你站到更高视角再判断。

还有一个更主动的用法：把已知 bug 或废弃方法扔给它，让它扫全仓库找同类模式，做预防性清理，而不是等后来人踩坑。

### 最佳实践

#### 先 Ask，再 Code

这是 OpenAI 在文档里反复强调的一点，它不像技巧，更像工程纪律。

大多数人拿到需求就直接扔给 Codex 写代码。写完发现方向不对、遗漏关键约束、改动范围比预期大三倍，然后返工。

他们的做法是：大改动先让 Codex 出实施计划，你确认、调整之后，再切到写代码。这两步不能省。文档里明确建议，单次任务最好控制在约 1 小时工作量、几百行代码以内，超出就拆。任务越小越稳，这不是保守，而是经验。

#### 把开发环境当成需要迭代的产品

Codex 能不能跑成功，和你的环境配置关系很大。

启动脚本不完整、环境变量没配好、网络访问权限没开，这些问题在你本地可能不明显，但会让 Codex 在执行任务时反复失败。你以为是模型不行，其实是环境在拖后腿。

OpenAI 的建议是把环境本身当成一个持续迭代对象。每次遇到构建报错，不要只解决这一次，顺手把修复方法写进环境配置里。下一次 Codex 就不会在同一个地方栽跟头。

#### 把 Prompt 写得像 GitHub Issue

“帮我重构一下这个模块”这种 Prompt，产出的结果大概率不是你想要的。

GitHub Issue 是有结构的：背景是什么、问题在哪、期望结果是什么、边界在哪。Prompt 也应该这样。

具体来说：把文件路径给出来，把涉及组件名写进去，把相关 diff 或文档片段贴上。如果有对标实现，直接说“按 X 模块的方式做”，比描述半天更准确。上下文越具体，Codex 跑偏的概率越小。

#### 把任务队列当成轻量 backlog

不是每件事都值得你当场做完。

工作中有大量“顺手可修”的事情：一个命名、一处明显可简化的逻辑、一个迟早要处理的技术债。以前你要么当场分心去做，要么记在脑子里然后忘掉。

更好的方式是：想到就丢进任务队列，让 Codex 先生成计划或初稿，不必立刻完成。你回到心流时再挑最有价值的做，或者直接 review 它已经准备好的内容。

#### 写一个 AGENTS.md

这是这份文档里最值得立刻去做的一件事。

Codex 没有记忆。你上次告诉它的命名约定、团队业务规则、不能碰的依赖版本，下次对话它都不知道。你要么重新交代，要么它按“通用常识”给你一个落不了地的答案。

解决方式是在仓库里维护一个 `AGENTS.md`：把命名规则、目录结构、业务背景、已知坑、常用命令都写进去。Codex 每次启动任务都会读这个文件，等于给它一份持久的项目说明书。

这件事成本很低，但长期收益很高：每次调用都更准、更稳、更少跑偏。

#### 用 Best-of-N 提升输出质量

遇到复杂或重要任务，不要只要一个答案。

让 Codex 一次给你三个不同方案，然后你来挑。你会发现三个方案里通常没有一个完美，但你可以把 A 的结构、B 的错误处理、C 的命名方式组合起来，得到比单次输出更好的结果。

这件事的本质是把“生成”和“决策”分开。它负责生成选项，你负责判断和取舍。

### 为什么值得认真对待提示工程

有一个问题值得单独讲：为什么写好 Prompt 比你想象中更重要？

它直接影响模型输出质量。同一个模型、同一个任务，Prompt 清楚和模糊，结果差距可能很大。你提供的上下文越完整、边界越清晰，输出越准，后续改动成本越低。

它能减少偏离和有害输出。模型不是在“理解”你，它是在补全你给它的输入。你说得越含糊，它越容易走向训练数据里的平均模式，而不是你的真实需求。

它让整个使用过程变得可预测。结构化 Prompt 可以复用。你今天用它完成一个任务，下次遇到类似场景，改几个变量就能继续用。积累够了，你就有了一套自己的 Prompt 工作流，而不是每次从零摸索。

它让模型真正成为你的工具，而不是随机建议机器。你越清楚自己要什么，Prompt 写得越准，模型越能稳定给你你想要的结果。

提示工程听起来很技术，实际上是一件很朴素的事：你越会表达自己的需求，AI 工具就越能帮到你。和人沟通如此，和模型沟通也是如此。

### 写在最后

OpenAI 在文档最后提到，Codex 还处于 research preview 阶段。

但他们也明确说，它已经在内部真实改变了工程交付节奏：更快完成原本没人愿意碰的收尾工作，更稳推进跨多个模块的大改动，更早发现本来要上线后才暴露的问题。

我读完这份文档最大的感受是：他们不是在用一个更聪明的工具，而是在用一套更好的工作方式。

工具是 Codex，但真正起作用的是“先 Ask 再 Code”的节奏、持续迭代环境的耐心、把上下文写清楚的习惯，以及在仓库里维护 `AGENTS.md` 的意识。

这些东西，换哪个工具都适用。
